# PRD: TokCard 核心体验重构（聚焦版）

## 1. 产品定位

TokCard 不应该再像一个“参数很多的卡片编辑器”，而应该像一个**专门展示 AI 战绩与项目身份的名片生成器**。

用户打开 TokCard 后，应该在很短时间内回答清楚三件事：

1. **你烧了多少 Token？**
2. **你在做什么项目？**
3. **你排第几，别人都在做什么？**

其他内容——主题、背景、比喻、文案、更多链接——都只能服务于这三个核心问题，不能抢戏。

---

## 2. 本次重构的核心决策

### 决策 A：创建页只保留一个主路径
默认先填写四项：
- Token 总量
- 项目名称
- 项目链接
- 项目一句话介绍

其余设置全部归入“高级个性化”。

### 决策 B：卡片视觉降噪
- 卡片种类不再追求多
- 背景图不再追求复杂
- 只保留**精选主题** + **精选纯色/渐变**
- 每个选项都要足够精美，宁缺毋滥

### 决策 C：项目介绍必须更有传播力
项目介绍不能只是“这是我的项目”。
它应该是：
- 一句话
- 简短
- 有记忆点
- 有野心
- 能吸引人点进去

### 决策 D：排名不是附属信息，而是主卖点之一
用户不仅要看到“自己用了多少 Token”，还要看到：
- 自己的排名 / 百分位
- 更强的人用了多少
- 榜上的人在做什么项目

### 决策 E：AI 文案统一走“短、狠、准”路线
AI 生成的内容必须：
- 短
- 不啰嗦
- 不像模板
- 有数字 / 比喻 / 力度
- 一眼能抓住用户

---

## 3. 目标

1. 创建页在 3 秒内让人明白：这是在做“Token + 项目 + 排名”名片
2. 新用户可以只填核心信息就完成一张能分享的卡片
3. 项目介绍成为卡片和分享页的重要信息，而不是附属链接
4. 排行榜从“单纯数字列表”升级为“谁最能打、都在做什么”的发现页
5. AI 生成文案更短、更强、更适合传播
6. 视觉系统更克制，减少花哨设置带来的决策疲劳

---

## 4. 设计原则

### 4.1 信息层级
始终保持同一顺序：
1. Token
2. 项目
3. 排名
4. 其他装饰信息

### 4.2 默认即最佳实践
用户第一次使用时，不应该被迫理解十几个设置项。
默认方案就应该足够好看、足够能发。

### 4.3 一张卡只讲一个主项目
允许保留多个项目链接，但界面和分享逻辑必须突出**一个主项目**。

### 4.4 精选优于海量
- 精选 4 个卡片预设，而不是 8 个以上
- 精选 5 个主题，而不是 10 个以上
- 精选 8 个背景色板，而不是复杂图像背景

### 4.5 AI 输出必须可直接使用
AI 生成的项目介绍、比喻、爆款文案，都应该接近“复制即可发”，而不是需要大量二次编辑。

---

## 5. 用户故事

### US-001：新增“主项目身份”数据结构
**描述**：作为 TokCard 用户，我希望卡片有一个明确的主项目信息，这样卡片、分享页和排行榜都能围绕同一个项目讲故事。

**验收标准：**
- [ ] `CardData` 新增主项目字段：项目名称、项目链接、项目一句话介绍
- [ ] 模板编码 / 解码、分享 payload、存储 payload 都能携带这三个字段
- [ ] 旧卡片链接在没有新字段时也能安全解码并使用默认值
- [ ] 现有多项目 `projects` 结构继续保留，但降级为次要信息
- [ ] Typecheck passes

### US-002：创建页重构为“核心信息优先”
**描述**：作为首次使用的用户，我希望一进入创建页就只看到最重要的信息输入，不被杂乱设置打断。

**验收标准：**
- [ ] 创建页首屏只默认展示四个核心字段：Token 总量、项目名称、项目链接、项目一句话介绍
- [ ] 现有头像、背景、主题、比喻、多项目、trust 等设置移动到折叠的“高级个性化”区域
- [ ] “高级个性化”默认关闭
- [ ] 核心区域的标题、间距、输入框尺寸明显强于其他区域
- [ ] 预览卡片在修改核心字段时实时更新
- [ ] Typecheck passes
- [ ] Verify in browser using dev-browser skill

### US-003：支持 AI 生成“项目一句话介绍”
**描述**：作为用户，我希望 TokCard 能根据我的项目链接或项目名称，生成一句很能打的项目介绍，帮助我更好地展示项目。

**验收标准：**
- [ ] 在“项目一句话介绍”输入框旁提供 AI 生成功能
- [ ] 如果填写了项目链接，优先读取页面 title / description 作为提示词上下文；读取失败时回退到项目名称
- [ ] 一次返回 3 个候选结果，供用户点击选择
- [ ] 中文结果每条不超过 24 个字，英文结果每条不超过 12 个单词
- [ ] 候选结果强调项目价值、速度、规模、结果或野心，不输出空泛套话
- [ ] 用户可以重新生成，并可手动编辑最终文案
- [ ] Typecheck passes
- [ ] Verify in browser using dev-browser skill

### US-004：重写 AI 文案策略，压缩比喻 / slogan / 爆款文案
**描述**：作为用户，我希望 AI 生成的比喻、slogan 和爆款文案更短、更有力、更适合直接传播。

**验收标准：**
- [ ] slogan 生成结果每次最多返回 3 条
- [ ] 中文 slogan 每条不超过 12 个字，英文 slogan 每条不超过 15 个单词
- [ ] 比喻生成结果每次最多返回 3 条，中文每条不超过 18 个字
- [ ] 爆款文案标题短、正文短，默认不输出冗余段落，hashtags 最多 2 个
- [ ] 提示词中使用用户的 Token 数、主项目名称、项目介绍、排名信号作为上下文
- [ ] 输出避免模板感、重复句式和空泛自嗨
- [ ] Typecheck passes
- [ ] Verify in browser using dev-browser skill

### US-005：将卡片预设缩减为 4 个精选模板
**描述**：作为用户，我希望卡片模板少而精，能够快速选到适合自己的风格，而不是在太多预设里犹豫。

**验收标准：**
- [ ] 当前卡片预设数量缩减为 4 个
- [ ] 4 个预设分别覆盖：战绩炫耀、项目发布、专业名片、社交传播
- [ ] 每个预设都围绕 Token、项目、排名三个重点配置默认文案和视觉风格
- [ ] 删除或隐藏现有不够聚焦的预设项
- [ ] 预设选择区文案更明确，用户能快速理解每个模板适合什么场景
- [ ] Typecheck passes
- [ ] Verify in browser using dev-browser skill

### US-006：用精选纯色 / 渐变替换复杂背景，并压缩可见主题数量
**描述**：作为用户，我希望卡片视觉更高级、更舒服，不需要复杂背景图，只需要少量精选配色。

**验收标准：**
- [ ] 用 8 个精选纯色 / 渐变背景替换当前图片型背景选项
- [ ] 默认流程中不再强调自定义图片背景
- [ ] 主题选择器可见项缩减为 5 个精选主题
- [ ] 每个背景 / 主题组合都经过对比度检查，保证文字可读
- [ ] 背景与主题名称更易理解，避免技术化命名
- [ ] Typecheck passes
- [ ] Verify in browser using dev-browser skill

### US-007：重排卡片与分享页的信息顺序，突出 Token → 项目 → 排名
**描述**：作为看到卡片的人，我希望第一眼就理解这张卡最重要的信息，而不是被装饰项和次要信息分散注意力。

**验收标准：**
- [ ] `CardRenderer` 中最显眼的信息是 Token 数量
- [ ] 主项目名称和一句话介绍在卡片中可见，不需要额外点击才能理解项目是什么
- [ ] 排名信息以徽章、百分位或 rank summary 的形式作为第三层重点出现
- [ ] 现有多项目链接仍可保留，但位置和权重低于主项目信息
- [ ] `SharedCardLanding` 的一行摘要和行动按钮也遵循 Token → 项目 → 排名的顺序
- [ ] Typecheck passes
- [ ] Verify in browser using dev-browser skill

### US-008：在创建与分享链路中强化“我的排名”反馈
**描述**：作为创建者，我想在生成卡片后快速知道自己的排名处于什么位置，并能一键跳去排行榜看自己和别人对比。

**验收标准：**
- [ ] 对可参与排名的卡片，在创建页导出 / 分享后显示“我的排名摘要”模块
- [ ] 摘要至少包含：全站排名或百分位、当前档位、跳转排行榜按钮
- [ ] 如果当前卡片已有可聚焦的 card id，跳转到 `/rank?focus=...`
- [ ] 分享页继续展示当前卡片的排名摘要，不要求用户再次搜索自己
- [ ] Typecheck passes
- [ ] Verify in browser using dev-browser skill

### US-009：把排行榜做成“看高手在做什么”的发现页
**描述**：作为普通用户，我希望排行榜不只是显示谁 token 高，还能清楚看到榜上用户在做什么项目。

**验收标准：**
- [ ] 排行榜列表默认展示：排名、用户名、Token 数、主项目名称、项目一句话介绍、模型 / 渠道标签
- [ ] 重点区域突出 Top 3 或当前焦点用户，便于快速对标
- [ ] 当用户通过 `focus` 进入排行榜时，焦点用户的信息卡更明显
- [ ] 现有复杂筛选项的层级下降，不抢列表主内容的注意力
- [ ] Typecheck passes
- [ ] Verify in browser using dev-browser skill

### US-010：首页与导航围绕“晒 Token / 讲项目 / 看排名”统一表达
**描述**：作为新用户，我希望首页一打开就知道 TokCard 到底解决什么问题，并有明确入口继续操作。

**验收标准：**
- [ ] 首页主标题和副标题明确传达三件事：晒 Token、讲项目、看排名
- [ ] 首页主 CTA 明确指向创建卡片，次 CTA 明确指向排行榜
- [ ] 首页示例卡、功能介绍和文案统一围绕三个核心价值，不再扩展无关卖点
- [ ] 导航层级更轻，减少分散注意力的入口
- [ ] Typecheck passes
- [ ] Verify in browser using dev-browser skill

## 6. 功能需求

- FR-1：系统必须支持一个“主项目身份”对象在卡片、分享和排行中复用
- FR-2：创建页必须将核心输入与高级个性化输入分层显示
- FR-3：系统必须支持基于项目链接 / 项目名生成一句项目介绍
- FR-4：系统必须压缩 AI 文案长度并提升传播性
- FR-5：系统必须减少可见模板、主题、背景数量，降低选择成本
- FR-6：卡片与分享页必须优先展示 Token、项目、排名三类信息
- FR-7：创建链路必须提供用户自己的排名反馈
- FR-8：排行榜必须同时展示“谁用了多少 Token”和“他们在做什么项目”
- FR-9：首页表达必须与产品核心价值完全一致

---

## 7. 非目标

- 不新增用户登录 / 账户系统
- 不做复杂社交 feed 或评论系统
- 不引入新的大模型供应商，只在现有 AI 能力上优化提示词和流程
- 不保留大量花哨背景图作为主打能力
- 不把 TokCard 做成通用设计工具或万能海报编辑器

---

## 8. 设计规范

### 8.1 信息权重
- Token 最大
- 项目次之
- 排名第三
- 其余信息为辅助

### 8.2 文案规范
- slogan：短，像标题
- 项目介绍：短，像推介语
- 爆款文案：短，像可直接复制的社交发帖内容
- 比喻：短，像记忆钩子

### 8.3 视觉规范
- 优先纯色和渐变
- 减少大面积复杂纹理
- 主题必须服务于内容清晰度
- 每个精选项都需要实际预览验证

---

## 9. 成功指标

- 新用户首次完成卡片所需步骤明显减少
- 更多用户在卡片中填写主项目信息，而不是只填 Token
- 排行榜页的点击深度和停留时长提升
- AI 生成文案被直接采用的比例提升
- 用户更容易理解 TokCard 的核心价值，而不是困惑于功能堆叠

---

## 10. 交付顺序建议

1. 主项目数据结构
2. 创建页核心信息重构
3. 项目一句话 AI 生成
4. AI 文案策略压缩
5. 精选模板
6. 精选主题 / 背景
7. 卡片与分享页重排
8. 我的排名反馈
9. 排行榜发现页优化
10. 首页与导航统一表达
